# Panel Review — BC Perf Observability Platform Umbrella Spec

**Date:** 2026-07-09
**Spec reviewed:** `docs/superpowers/specs/2026-07-07-bc-perf-observability-platform-design.md` (as committed in 3ea5369)
**Method:** six independent review agents, one lens each: performance/scalability, security, BC-domain realism, devil's advocate, future/strategy, design coherence. Full verbatim reports below, synthesis first.

---

## Synthesis (convergent findings)

1. **SaaS production has zero capture path** (BC, devil, future — fatal/critical). Inside track blocked on Microsoft; outside track structurally sandbox-only (snapshot debugging unsupported on SaaS production by policy). Hedges under our control: codeunit 1924 self-profiling canary (Job Queue + synthetic workload + auto-ship, works today) and Application Insights telemetry as a third ingestion source / trigger layer.
2. **Encryption-at-rest contradicts reprocessing** (perf, design, sec). Tenant-public-key encryption + plaintext deletion vs promised fleet reprocessing, lifecycle scans, heat-map aggregation. Fix: explicit two-tier data model — server-readable derived data vs tenant-opaque blobs.
3. **Merged-workspace graphs contradict build-on-registration** (perf, design). App combination only known at profile arrival. Fix: combination-graph key (hash of sorted app-version set), lazy build behind queue.
4. **Tenant isolation claimed but absent** (sec — 4 criticals). Global shared secret, spoofable `x-tenant-id`, tenant-less registry/graph cache, cross-tenant reads, plus unauthenticated `/api/record-next-batch` and `/api/analyze` in today's image.
5. **Lifecycle engine under-specified** (design, future, devil, bc). Fingerprint undefined for profile-only path; rename severs identity; "absent N runs" undefined across capture kinds; §4 ack-gating contradicts §5 outbox; baselines must be keyed (tenant, schedule, captureKind) and version-stamped; flapping on sampling noise is the top trust risk.
6. **ir-json ingest unbounded** (perf — measured 12.8x vs .alcpuprofile, ~537 B/invocation). Needs compression, size budget, bounded captures, and a designed durable analysis queue (current handler analyzes synchronously in-request).
7. **Phase reorder** (future, devil). Lifecycle before server-side fusion; fusion ships CLI-first (~4,100 lines already on feat/alsem-fusion); sub-project 3 is a dependency watch, not a workstream; source custody blocks ISV adoption — accept prebuilt artifacts instead of raw source.
8. **Agentic ops layer shrinks** (devil, sec). Three of four duties are deterministic; prompt-injection loop via finding text into a write-capable agent; keep triage only, read-mostly, human confirm on outward writes.
9. **Missing pages:** cross-repo contracts/compat policy (3 reviewers), retention/GDPR/PII (3 reviewers), risk register (future).

## Review 1 — Performance/Scalability (rev-perf)

Spec reviewed: U:\Git\al-perf\docs\superpowers\specs\2026-07-07-bc-perf-observability-platform-design.md (read-only review; no files edited). Findings are grounded in the current code (web/handlers/ingest.ts, web/server.ts, src/history/store.ts, src/semantic/engine-runner.ts) and one hard measurement: I ran bc-mdc-converter's release binary on its own instr.mdc.zip fixture (11,285 invocations) — the ir-json output is 6.07 MB vs 474 KB for the equivalent .alcpuprofile of the same capture: 12.8x larger, ~537 bytes/invocation.

### Findings

**1. [CRITICAL] ir-json payload size is unaccounted for — §1 "Ingest contract change"**
Measured 12.8x size vs .alcpuprofile at ~537 B/invocation. Instrumentation records every AL invocation, so a busy production session over minutes reaches 10^5–10^7 invocations, i.e. 50 MB–5 GB of uncompressed JSON. The ir-json format spec (bc-mdc-converter, 2026-07-06-ir-json-format-design.md §8) explicitly declares "no compression, no NDJSON, file size not a constraint" — fine for a local CLI, but the umbrella spec adopts it as a network ingest payload without comment. The current server caps bodies at 100 MB (web/server.ts:45), which at 537 B/invocation means ~190k invocations — easily exceeded — and a single JSON.parse of a body that size blocks the Bun event loop for seconds with 5–10x transient memory amplification.
*Recommendation:* mandate gzip/zstd transfer encoding on /api/ingest (ir-json is highly repetitive; expect 10–20x), set an explicit invocation-count/size budget in the contract, and require either NDJSON-streamable parsing or a capture-duration/session-filter guardrail in the orchestrator so captures are bounded at the source. State the expected payload envelope (P50/P99) in the spec.

**2. [CRITICAL] "Queue for analysis" is one phrase carrying the whole scalability story — §2 "Ingest" flow**
The spec's flow "ingest → store → queue for analysis → store result" is the right shape, but the queue is completely unspecified — and the current implementation is the inverse: web/handlers/ingest.ts:106-114 runs analyzeProfile synchronously inside the request handler, then returns 202. Analysis is CPU-bound JavaScript (parse, call-tree build, detectors) plus AI deep analysis (the web server always runs deep analysis when ANTHROPIC_API_KEY is set — tens of seconds) plus fusion (alsem subprocess, 60 s default timeout in src/semantic/engine-runner.ts:83). On a single Bun process, top-of-the-hour Job Queue alignment across tenants produces bursts of concurrent ingests that serialize on the event loop and time out. Sink transitions committing "only after sink acknowledgement" (§5) also require the queue to be persistent, not in-memory.
*Recommendation:* specify the execution model explicitly: persist raw payload + return 202 immediately; SQLite-backed durable job queue; N worker processes (or Bun Workers) for CPU-bound analysis with a concurrency cap; crash-recovery = re-queue incomplete jobs. Fits the solo-dev/Docker posture and is the single biggest determinant of 10x survival.

**3. [CRITICAL] Encryption-at-rest contradicts reprocessing, resolution scans, and fleet aggregation — §2 vs §5**
In the POC path the spec keeps for SaaS, both the profile and the analysis result are RSA/AES-encrypted with the tenant's public key and the plaintext deleted (web/handlers/ingest.ts:118-150) — the server cannot read them afterward. That directly contradicts §5 "the profile is stored regardless and is reanalyzable after a fix — this also enables fleet reprocessing after detector upgrades," and undermines the lifecycle engine (absent-N-runs needs past results), the fleet heat map (§3.6 aggregate queries over fusion output), and agentic post-fix verification (§4). Only the plaintext manifest.json/metrics.json extracts survive as a queryable surface today.
*Recommendation:* define the plaintext queryable extract explicitly — which finding fingerprints, per-routine metrics, and fusion attributions live unencrypted in SQLite (derived metadata, not source IP) vs what stays in the encrypted blob. If fleet reprocessing is a SaaS requirement, the model needs a server-held key option; otherwise scope that requirement to self-host only.

**4. [MAJOR] Multi-app merged-workspace graphs break the "build on registration" invariant — §2 "Source registry" + "Graph cache invariants"**
The registry is keyed {appId, version} and builds on registration, but fusion needs a graph "built over the merged workspace" of all apps a profile spans. The merge set (base app version × ISV versions) is only known when a profile arrives — so merged builds happen on first profile after all, exactly what the invariant forbids as too expensive. Merge sets are also combinatorial across a fleet (every tenant runs a different version mix), each combination being a separate expensive build and a separate large cached artifact competing for LRU space.
*Recommendation:* either (a) key merged-graph cache entries by the sorted hash of the app-version set and accept first-profile builds behind the build queue with a "fusion pending" flag on early results, or (b) redesign toward per-app graphs composed at fusion time (al-call-hierarchy already resolves cross-app edges via .app symbols — investigate composition without full re-analysis). The spec must pick one; right now it silently assumes both.

**5. [MAJOR] No concurrency cap on graph builds — §2 "build asynchronously"**
POST /api/sources is CI-triggered; release days fan out multiple simultaneous registrations, each spawning aldump + alsem analyze over Base-App-scale workspaces (whole-program resolution — CPU/memory-heavy Rust processes, plausibly GBs RSS each). Several concurrent builds in one Docker container will OOM it or starve the analysis path.
*Recommendation:* dedicated build queue with concurrency 1–2, dedup by content hash (the in-flight-promise pattern in engine-runner.ts is the right prior art), and a documented memory ceiling for Docker sizing.

**6. [MAJOR] The regression-baseline data model is missing, and it's the biggest table in the system — §4 "Trigger rules"**
"Routine self-time +X% vs a rolling baseline" requires per-routine, per-run metrics persisted. Nothing stores those today: the history store keeps only the top-5 hotspots per entry (src/history/store.ts:68) and ingest's metrics.json keeps summary counters. At 10 tenants × 5 schedules × 1,000 routines × 365 daily runs ≈ 18M rows/year — fine for SQLite with a designed schema, indices on (tenant, schedule, routineId, runAt), and retention/rollup, none of which the spec mentions.
*Recommendation:* add a data-model section: per-routine metrics table, index plan, retention (e.g., raw rows 90 days, daily rollups after). Also note the existing history store reads and JSON-parses every file on every query (store.ts:89-103) — at platform scale it should be folded into the same SQLite DB, not kept "alongside" it (§4 says "SQLite alongside the existing history store", but the existing store is JSON-files, not SQLite).

**7. [MAJOR] AI analysis cost scales linearly with ingest volume — §3 "AI deep-analysis upgrade"**
The web server currently runs deep AI analysis on every profile when the key is set. Under continuous scheduled capture that becomes the dominant cost and latency term: thousands of Opus/Sonnet deep-analysis calls per day at 10x scale, mostly re-analyzing an unchanged steady state. The spec never states when AI runs.
*Recommendation:* make AI analysis event-driven: only on new or regressed fingerprints, or needs-triage. The deterministic pipeline handles the steady state. Stating this policy changes the cost model by orders of magnitude.

**8. [MAJOR] Fleet heat map needs persisted, queryable fusion output — §3.6**
"Many profiles × one graph → runtime heat" is an aggregate query. If fusion attributions live only inside per-profile AnalysisResult JSON (encrypted, per finding 3), the heat map requires decrypt-and-scan of every profile per query — non-viable. The per-profile fusion join itself is fine (map-based after commit c942e77); the aggregation layer is unspecified.
*Recommendation:* persist fusion attributions (routine/edge fingerprint, cost, profile ref, tenant, schedule) as SQLite rows at analysis time so the heat map is a GROUP BY; size this table together with finding 6's.

**9. [MINOR] No data-retention story anywhere — whole spec**
Profiles (encrypted blobs), analysis results, finding occurrences, history entries, and registry artifacts (source zips + .alpackages symbols — Base App symbols are hundreds of MB per version) accumulate indefinitely. Only the graph cache has eviction.
*Recommendation:* one retention section: per-tenant quotas, TTLs per artifact class, registry pruning (keep last K versions per app).

**10. [MINOR] SQLite write-path discipline once workers exist — §4**
Fingerprint upserts, occurrence inserts, outbox writes, and absent-N-runs scans share one SQLite file. Fine at this scale if WAL mode + busy timeout are specified and writes are funneled through one process (the job-queue owner). The absent-N-runs scan is cheap with an index on (tenant, schedule, state, lastSeenRun) — worth stating so it isn't a full-table scan per run. Also: the existing stats.json read-modify-write (web/server.ts:82-88) loses updates under concurrent ingest.

**11. [MINOR] Sink rate limits during storms — §4 "Sink adapters"**
GitHub PATs get ~5,000 req/hr plus aggressive secondary limits on issue creation. A fleet-wide regression (e.g., a bad BC platform update) can generate hundreds of findings at once. Quiet hours help, but the outbox should specify per-sink rate limiting and collapse-to-epic before the agentic layer exists (the spec currently assigns escalation only to the agent).

**12. [MINOR] Ingest burst alignment — §1 "Inside track"**
BC Job Queues fire clock-aligned; N tenants ship at the top of the hour simultaneously. With finding 2 fixed this is absorbed; without it, it's the trigger for the failure mode. Add one line: the ingest request path must be I/O-only and O(payload), plus optional jitter in the auto-ship scheduler.

### Verdict

The architecture direction is sound — hub platform, store-first ingest, subprocess-isolated semantic engine, deterministic pipeline with AI on top — and the graceful-degradation discipline is genuinely good. But as specced, the design hand-waves exactly the things that decide whether it survives its own success. What MUST change before sub-project specs are cut: (1) the ir-json ingest contract needs a size budget, compression, and a bounded-capture guardrail — a measured 12.8x, unbounded-growth payload cannot ride an endpoint with a 100 MB cap and a single-threaded parse; (2) the analysis queue must be promoted from a phrase to a designed component (durable, worker-backed, concurrency-capped, crash-recoverable), since every latency and burst property of the platform hangs on it; and (3) the SaaS encryption-at-rest model must be reconciled with reprocessing, lifecycle scans, and fleet aggregation by explicitly defining the plaintext derived-data surface — as written, §2 and §5 promise mutually exclusive things. The multi-app graph-cache keying (finding 4) is the fourth item to resolve at umbrella level rather than letting sub-project 4 discover it mid-implementation. Findings 6–8 are data-model gaps that belong in sub-project 5's spec but should be named in the umbrella now.

## Review 2 — Security (rev-sec)

Findings are grounded against the actual implementation: web/handlers/ingest.ts, web/handlers/tenants.ts, web/handlers/profiles.ts, web/poc-secret.ts, web/crypto.ts, src/source/zip-extractor.ts, web/server.ts, src/semantic/engine-runner.ts, plus al-perf-bc (bearer secret in IsolatedStorage — good) and bc-dev-mcp (BC_DEV_USER/BC_DEV_PASSWORD env credentials, plain http:// permitted).

### Authentication & tenant isolation

1. (CRITICAL — §2 Ingest, §5 Security) "Bearer per tenant" does not exist; it is one global shared secret. The spec's tenant-isolation story rests on "bearer per tenant (exists)", but handleIngest authenticates every request against the single process-wide AL_PERF_POC_SECRET (web/poc-secret.ts), and tenant identity comes from the caller-supplied x-tenant-id header, never bound to the token. Any secret holder can write to any tenant. Recommendation: issue a distinct high-entropy credential per tenant (store a hash), derive tenant from the credential server-side, and mark the current model as POC-only in the spec.

2. (CRITICAL — §1 Capture contract, §4 Lifecycle) Spoofed uploads poison baselines and drive external side effects. Because tenant is an unauthenticated header (finding 1) and the lifecycle engine computes regression deltas against rolling baselines and dedups by fingerprint, one leaked credential lets an attacker inject fabricated profiles under a victim tenant — shifting baselines to mask or fabricate regressions, and (via §4 sink rules) filing or mutating GitHub Issues / ADO Work Items in the victim's tracker. Recommendation: bind tenant to credential; authenticate capture provenance (the auto-ship RSA material can sign, not just encrypt); treat baseline mutations as audited security events.

3. (MAJOR — §2 Source registry) Tenant registration is self-service under the same secret — squatting and key substitution. /api/tenants/register gates on the same global secret, first-write-wins. The registered RSA public key is what ingest encrypts bundles to, so anyone with the secret can pre-register a victim tenant code with an attacker-controlled key (making future at-rest ciphertext attacker-decryptable) or squat codes. Registration is the trust anchor for the whole at-rest scheme. Recommendation: separate the admin registration secret from the ingest client secret; require an out-of-band admin action to bind tenant code to public key; make key rotation an explicit audited operation.

4. (MAJOR — §5 Security) Cross-tenant reads via GET /api/profiles/:id. handleGetProfile authorizes with the same global secret and takes tenant from a query param, returning the wrapped key + ciphertext and plaintext manifest/metrics. Any credential holder can enumerate and download any tenant's stored bundles; metadata and capture volume leak freely even where payloads stay RSA-protected. Recommendation: authorize reads by per-tenant credential, never a free query param; add an access audit log.

### Source-zip handling & graph builder

5. (MAJOR — §2 Source registry, §3 Fusion) The registry's zip surface exceeds what the extractor safely handles, and feeds untrusted archives into external parsers. extractCompanionZip is a hand-rolled ZIP reader (no data descriptors, no Zip64) that extracts only *.al and applies zip-slip guards only to those entries — but /api/sources must also accept app.json and binary .alpackages symbols, and engine-runner then spawns alsem/aldump over the extracted directory. .app files are containers parsed by that third-party toolchain, i.e. an attacker-controlled parser-attack surface reached from an upload. Recommendation: use a hardened unzip library; apply the slip guard to every entry; extract into a per-request jailed dir and pass only that to the engines; sandbox the alsem/aldump subprocesses (resource limits, no network, least-privilege user).

6. (MINOR — §2 Source registry) Decompression-bomb defense is partial. Per-entry (50MB) and total (500MB) caps exist and the inflate stream is bounded — good — but there is no entry-count cap and the whole archive is buffered in memory before processing; 100MB upload can expand to 500MB RAM per request. Recommendation: cap entry count, stream the archive, bound concurrent extractions.

### Multi-tenant data separation ("product both ways")

7. (CRITICAL — §2 Registry/graph cache, §3 Fusion) Registry and graph cache have no tenant dimension — cross-tenant source disclosure and fusion poisoning. The design keys the registry by {appId, version} and the graph cache by content hash with LRU eviction. Two tenants sharing an appId/version collide: tenant B's profiles fuse against tenant A's registered source (cross-tenant IP disclosure of exactly the customer IP §5 says to protect), and a malicious tenant who registers a common appId/version first poisons every other tenant's fusion output. The "nearest-lower version fallback" widens this. Filesystem paths are tenant-scoped (storage.ts is sound) but registry, graph cache, and the SQLite finding store are shared. Recommendation: make tenant a first-class key for registry, cache, and finding store; if sharing identical public base-app graphs is desired, gate it on content hash plus signed publisher identity, never on an assertable appId/version.

### At-rest protection & retention

8. (MAJOR — §2 Ingest, §5 Security) "Only ciphertext at rest" is contradicted twice. Ingest deletes the plaintext profile but writes manifest.json and metrics.json in clear (tenant tag, activity/schedule IDs, SQL/HTTP counts, health score). More fundamentally, bundles are encrypted to the tenant's public key so the server cannot read them — yet §4 requires the server and agent to read findings (source snippets, routine names) to file issues and triage, so that data lives server-readable in SQLite. The spec never reconciles "server can't read it" with "server files it externally". Recommendation: define explicit tiers — server-readable (findings, metrics; protect with server-held KMS/envelope encryption) vs tenant-opaque (raw profiles/source; tenant-key encrypted) — and encrypt or declassify the metadata sidecars deliberately.

9. (MAJOR — §5 Security) No retention, deletion, or erasure story for customer IP. Only debug captures have a retention (7 days). Ingested profiles, source, graphs, and findings accumulate indefinitely with no delete API, no cache TTL, no residency statement — a compliance and breach-blast-radius problem for hosted SaaS. Recommendation: per-tenant retention windows, an erasure API, TTLs on graph/registry caches, documented residency. Also note the debug-capture path itself: AL_PERF_DEBUG=1 writes every request's full profile + source zip to disk without consent — the spec should bring that path under the same policy.

10. (MINOR — §2 Ingest, web/crypto.ts) Dated primitives, no rotation path. RSA-OAEP-SHA1 + AES-256-CBC with encrypt-then-HMAC (correctly implemented, keys correctly separated) — not broken, but legacy defaults, and keyVersion is hardcoded "1" with no rotation procedure in the design. Recommendation: move to OAEP-SHA-256/AES-GCM when the BC-side AL crypto allows; specify key rotation now while the format is young.

### Secrets

11. (MAJOR — §5 Security) Sink PAT and BC-credential management is hand-waved. "Env/config secrets or encrypted at rest" doesn't answer how per-tenant PATs the server must actively use are protected (they can't be tenant-key encrypted); no secret backend is named; no least-privilege scoping is specified (GitHub PAT should be issues-only, ADO work-items-only). The capture orchestrator reuses bc-dev-mcp, which takes BC Basic-auth credentials from env vars and accepts plain http:// server URLs — privileged snapshot-debugger credentials can traverse cleartext. Recommendation: name a secret backend (KMS/Key Vault); mandate minimal PAT scopes per sink; require TLS for BC connections outside localhost; scope the capture account to profiler-only permissions.

### Sink abuse & agentic ops

12. (CRITICAL — §4 Agentic ops) Indirect prompt injection into a write-capable autonomous agent. Finding text (routine/object names, source snippets, AI narrative) is influenceable by whoever controls an uploaded profile or source zip. §4 flows that text into Issue/Work Item bodies, and the scheduled agent reads findings and issues back and acts with write tools (findings_update, report_file, sink adapters, epic escalation). That closes a loop: attacker-controlled content → issue → agent reads → agent acts externally. Recommendation: treat all finding/issue text as data, never instructions (delimited non-instruction context in agent prompts); allow-list the agent's tools and keep them read-mostly; require human confirmation for outward/irreversible actions; add per-run spend and rate caps.

13. (MAJOR — §4 Sink adapters) Markdown/template injection into trackers. Unsanitized finding text in issue bodies enables @mentions, link/image injection, task-list manipulation, and ADO field/area-path manipulation wherever tenant-influenced strings are interpolated. Recommendation: escape all interpolated fields, render finding text in fenced blocks, strip mention/directive syntax, validate labels/area paths against per-tenant allow-lists.

14. (MAJOR — §4 Agentic ops) Blast radius and audit gaps. The agent works cross-tenant ("trends across tenants"), holds write tools, and report_file has no stated path scope. The spec claims the audit trail is "identical" because writes go through the same sinks, but the agent's decisions/reasoning and reads are not captured, and cross-tenant reach means one bad run touches everyone. Recommendation: scope runs per-tenant where possible; jail report_file output; log every agent tool call with inputs/outputs; add a kill-switch and per-run budget.

### Omissions a v1 review would flag

15. (MAJOR — web/server.ts, absent from spec) /api/record-next-batch ships enabled and unauthenticated. It is gated only on NODE_ENV === "production", and the Dockerfile never sets NODE_ENV — in the published image any anonymous POST arms recording and the next customer's batch profiles are persisted to test/fixtures/batch-recorded. Recommendation: remove it from production builds or require the admin credential; set NODE_ENV=production in the Dockerfile regardless. Related: /api/analyze and /api/analyze-batch are fully unauthenticated today and trigger paid Anthropic API calls per upload when ANTHROPIC_API_KEY is set — an anonymous LLM cost-amplification vector on the hosted instance; the spec should state the auth posture for every /api endpoint, not just /api/ingest.

16. (MAJOR — §5 Security, deployment) No transport-security or DoS/rate-limit requirement. TLS is implied by the proxy but never required for self-host, so bearer tokens can cross plaintext HTTP; there are no per-tenant rate limits or quotas, and each ingest runs full (potentially AI) analysis. Recommendation: mandate TLS off-localhost; per-tenant rate limits and upload quotas; AI analysis opt-in with a spend ceiling; cap profiles[] per batch.

17. (MINOR — handlers) Error responses leak internal detail (detail: String(err) in ingest and analyze paths); CSP allows style-src 'unsafe-inline'; tenant-register compares the secret with a non-timing-safe !==. Recommendation: opaque error codes to clients with server-side logs; tighten CSP; use the existing timingSafeEqual helper everywhere.

### Verdict

The foundational hygiene is better than typical for a solo-dev POC — storage.ts path guards, encrypt-then-MAC, decompression caps, and subprocess reaping in engine-runner are genuinely careful work. But the design's central security claim, tenant isolation via "bearer per tenant (exists)", is not true in the code: one global shared secret authenticates all tenants and admin registration, and tenant identity is an unauthenticated header. That single gap cascades into cross-tenant read/write, baseline poisoning, and key substitution — and the design then stacks a tenant-less registry/graph cache and a write-capable cross-tenant autonomous agent on top, converting an auth weakness into cross-tenant IP disclosure and an indirect-prompt-injection loop with external side effects. Acceptable as a labelled POC for trusted single-tenant self-host; not yet a safe multi-tenant SaaS. Release-blockers for the hosted "product both ways" posture: findings 1, 2, 7, and 12, with 3, 4, 5, 8, and 14 close behind.

## Review 3 — BC Domain Realism (rev-bc)

Grounded against: the spec, U:\Git\al-perf-bc\MicrosoftRequest.md + README, U:\Git\bc-dev-mcp\docs\superpowers\specs\2026-07-03-snapshot-capture-design.md and 2026-07-04-instrumentation-profiling-design.md, U:\Git\al-call-hierarchy\README.md, plus the established BC29 artifact facts (system tables 2000000266/2000000265 OnPrem, pages 1931/1933 blocked, codeunit 1924 Cloud-accessible, scheduler sampling-only 50/100/150 ms).

### Findings

**1. CRITICAL — No capture path reaches SaaS production, which is where the observability promise matters most.** (Goal, Architecture diagram, §1)
The inside track is blocked on Microsoft opening OnPrem-scoped platform objects (may never happen), and the outside track can structurally never reach SaaS production: snapshot/instrumentation capture rides the developer endpoint, and Microsoft supports snapshot debugging only on online sandbox environments — production online is excluded by policy, not by missing auth work. "Entra ID for SaaS sandboxes later" does not change that ceiling. As written, the platform's headline — continuous performance observability including hosted SaaS customers — has zero data sources for the most common BC deployment model, and the architecture diagram's "SaaS-viable" label on the inside track is not true today. Recommendation: state the SaaS-production capture matrix explicitly (works today / after Microsoft / never), reframe the outside track as a depth tool for OnPrem, containers, and SaaS sandboxes, and add the two paths that do work in SaaS production today (findings 2 and 4).

**2. MAJOR — The spec omits the inside-track capture that works in SaaS today: codeunit 1924 self-profiling of scheduled synthetic workloads.** (§1 inside track, Phasing #3)
Codeunit 1924 "Sampling Performance Profiler" is Cloud-accessible and profiles the current session. A Job Queue entry can start profiling, execute a representative workload (post a synthetic order, run a report, exercise the ISV's hot paths), stop, and ship the .alcpuprofile through the existing auto-ship plumbing — a canary that needs nothing from Microsoft. It cannot observe real user sessions (that is what the scope request is for), but it gives sub-project 3 a deliverable even if Microsoft declines. Recommendation: add "self-profiled scheduled workload" as an explicit inside-track mode and make it al-perf-bc's default SaaS behavior, with scheduler-based fleet capture as the upgrade if scope opens.

**3. MAJOR — The Microsoft ask targets the wrong layer; calibrate expectations and timeline.** (Component inventory, §1, Phasing #3)
The spec and MicrosoftRequest.md say "tables 1924/1932," but the BC29 artifact check shows the data lives in platform system tables 2000000266 "Performance Profiles" / 2000000265 "Performance Profile Scheduler" — owned by the server team, not fixable by a BCApps System Application PR alone. The realistic outcome is not a scope flip but a permission-gated facade module (the request doc wisely says "any mechanism works"), and Microsoft will treat it as a privacy decision: scheduled profiles expose what named users did (User Name, session, activity), tenant-wide. Realistic timeline is a release wave or more (BC30+), and "no" is a live outcome. Recommendation: pursue via a BCApps GitHub issue, aka.ms/BCIdeas, and partner channels in parallel; design sub-project 3 so nothing idles waiting (finding 2); expect any granted facade to carry an admin-consent gate that al-perf-bc must surface in UX.

**4. MAJOR — Application Insights telemetry is the missing third ingestion source, and it is the only fleet-wide production-SaaS signal available today.** (§1, Architecture)
BC emits long-running SQL (RT0005), long-running AL method (RT0018), web-service latency, page-view, report, and job-queue telemetry to App Insights with zero Microsoft asks — and an ISV gets app-level telemetry for their own extension by setting applicationInsightsConnectionString in app.json, without per-customer setup. Telemetry lacks call trees, but it is the natural trigger layer: recurring RT0018 on a method opens a lifecycle finding, which schedules a targeted deep capture (scheduler OnPrem, sandbox instrumentation, or a cu-1924 canary) to get the profile. The lifecycle engine's fingerprint/baseline machinery fits telemetry findings almost unchanged. Recommendation: add telemetry ingestion (KQL export or App Insights REST) as a capture-layer track, or record an explicit non-goal with rationale. This is the first thing a veteran BC architect flags.

**5. MAJOR — Fusion outputs 1–2 over-promise on sampled profiles; 100 ms sampling yields very few samples per activity.** (§3 outputs 1, 2)
A typical profiled activity (0.5–5 s) at 50/100/150 ms intervals yields roughly 5–50 samples; "yours is 80% of that fan-out cost" is noise at that resolution, and short-lived subscribers are frequently invisible or misattributed. Hit counts in a sampling .alcpuprofile are sample counts, not call counts. Recommendation: gate per-edge percentage claims behind a minimum-sample threshold and attach confidence markers to fusion sections computed from sampling; make fleet aggregation (output 6) the flagship for the sampled inside track, and reserve precise blame percentages and edge counts for ir-json — the spec already restricts output 5 correctly; extend that discipline to 1 and 2.

**6. MAJOR — Instrumentation probe effect and capture-size limits are unaddressed.** (§1 outside track, §3 output 5)
Deterministic every-call instrumentation systematically inflates small, frequently-called methods relative to production behavior — "what-if simulation with real numbers" gets real counts but distorted times; the bc-dev-mcp design itself notes instrumentation self-time is deterministic call-time, not statistical. Full-verbosity recordings are also large (11,285 .mdc files for a short Cronus session); a cron capture of a long OnPrem production session risks very large recordings and server memory pressure. Recommendation: hard duration/size caps and off-peak scheduling guidance in the orchestrator config; fusion should annotate instrumentation timings as overhead-skewed (counts trustworthy, ratios of tiny methods not) and prefer counts-times-sampled-time hybrids in what-if math where both exist.

**7. MAJOR — Outside-track operational realism: the dev endpoint is usually disabled on production OnPrem, and attach semantics don't fit blind cron scheduling.** (§1 outside track)
Production OnPrem service tiers typically have the developer endpoint off as security hardening — asking customers to enable it for a profiling daemon is a security-review conversation, not a config line. Snapshot attach binds "next matching session" for a user/client-type filter: a scheduled capture may bind nothing during its window or bind an unrepresentative session, and the affinity-cookie/session lifecycle means captures die if the daemon restarts mid-flight. Assume concurrent snapshot-session limits per environment until verified. Recommendation: preflight via snapshotendpointmetadata, no-match timeouts with "0 sessions captured" as a first-class non-error outcome, capture state persisted across restarts, and deployment docs that treat dev-endpoint enablement as a customer security decision. Also note sandbox hardware differs from production — sandbox instrumentation timings are structural signals, not production benchmarks.

**8. MAJOR — Privacy/GDPR handling is thinner than the data warrants.** (§5 Security)
Scheduled profiles carry User Name, session IDs, and activity descriptions; raw .mdc retained "for manual recovery" comes from a debugging mechanism and may contain more than timings depending on verbosity. Hosted SaaS mode makes you a data processor for EU customers: DPA, retention policy, and data residency are table stakes, and encryption-at-rest (which the spec has) is not a substitute. Recommendation: PII minimization at capture (strip or pseudonymize user identifiers before ship, config-controlled), a documented retention/purge policy per store (profiles, raw .mdc, findings), and an audit of what Full-verbosity .mdc actually contains before committing to retain it server-side.

**9. MAJOR — al-perf-bc packaging and distribution constraints are real and unmentioned.** (Component inventory, §1, Phasing #3)
A single app cannot target Cloud while containing page extensions on 1931/1933 (OnPrem source tables) — al-perf-bc needs an app split (Cloud app: page 1911, cu-1924 flows, auto-ship; OnPrem companion: scheduler pages) or preprocessor builds. For PTE distribution, "Allow HttpClient Requests" defaults off in production — every auto-ship onboarding must include that toggle. For AppSource: object ID 70500 sits in the PTE range (AppSource requires 100,000+ plus a registered affix), and an app transmitting profile data externally must declare the dependency, ship a privacy statement, and will get validation scrutiny on consent; the hardcoded Locked-label endpoint would need to become setup. Recommendation: sub-project 3 should explicitly pick the distribution model (PTE vs AppSource), since it drives ID ranges, consent UX, and certification effort.

**10. MAJOR — Version churn is monthly, not semiannual; the registry design will thrash on Microsoft apps.** (§2 source registry, graph cache)
SaaS tenants take monthly minor updates on staggered update rings, so the fleet's base-app version is a moving distribution, not two versions a year. Keying {appId, version} with tenant-isolated storage means rebuilding and storing near-identical multi-thousand-object base-app graphs many times over. Recommendation: two-tier cache — a global cross-tenant tier for Microsoft-published apps keyed by artifact/content hash (sourced automatically from BC artifacts or BCApps rather than customer CI pushes), and tenant-isolated storage only for customer/ISV apps. Nearest-lower fallback is safe within a minor's patch levels; flag it loudly across minor boundaries.

**11. MAJOR — Regression baselines must be version-aware or monthly updates will generate false regressions.** (§4 trigger rules)
A routine's self-time can shift because Microsoft changed the base app or platform in a monthly update, with zero customer code change. A rolling baseline spanning an update boundary fires "+X% regression" findings blaming the wrong party and erodes trust in the lifecycle engine fast. Recommendation: stamp every history entry with platform + base-app + ISV-app versions (ingest metadata already carries appVersions), segment or reset baselines at version boundaries, and have the lifecycle engine annotate transitions coinciding with a version change as "environment changed" rather than "regressed."

**12. MINOR — Fusion coverage for base-app hotspots depends on what the graph recovers from symbols alone; verify before promising event-chain blame there.** (§3)
Real profiles are dominated by base-app frames (posting routines etc.). al-call-hierarchy resolves external .app packages via SymbolReference.json — signatures and attributes, not bodies — so EventFlow/ImplicitTrigger edges internal to Microsoft apps may be incomplete unless base-app source is registered. Recommendation: state in §2 whether the registry is expected to hold Microsoft app source (available per version from BC artifacts) and measure how much of the event graph symbols alone recover; scope fusion output 1 accordingly.

**13. MINOR — BCPT is a natural workload driver the spec doesn't mention.** (§1, Phasing #2)
The Performance Toolkit runs repeatable synthetic workloads in sandboxes — exactly what the orchestrator needs on the other side of an instrumentation attach, and what a cu-1924 canary should execute. Recommendation: one paragraph in the orchestrator sub-project ("a capture job may trigger or coincide with a BCPT suite run"), giving deterministic before/after profiles for gate checks and the agentic layer's post-fix verification duty.

**14. MINOR — The docker-compose E2E assumes a BC container; BC server containers are Windows-only.** (§5 Testing)
"docker compose — BC container + orchestrator + server" won't run on Linux CI; BC containers need a Windows host with process/Hyper-V isolation, which is painful on hosted runners. Recommendation: mark the E2E as local / self-hosted-Windows-runner only and keep CI at the mocked/contract level the spec already defines.

**15. MINOR — Verify scheduled-profile retention on the platform side.** (§1 inside track)
The auto-ship cadence only matters relative to how long the platform keeps rows in table 2000000266 — if the server purges scheduled profiles on a short window, a stalled Job Queue silently loses fleet data. Recommendation: verify the retention window during sub-project 3's artifact check and size the ship cadence plus a "gap detected" health signal accordingly.

### Verdict

The architecture is sound where it touches things you control — hub platform, ir-json as interchange, graceful degradation, fusion on routine identity, and the lifecycle engine are well-conceived and grounded in components that demonstrably exist. Where it touches the BC platform, the spec is optimistic in one load-bearing place: both capture tracks dead-end short of SaaS production (inside blocked on a Microsoft decision aimed at platform-owned tables, outside structurally sandbox-only), yet the goal statement and architecture diagram read as if fleet observability for SaaS customers is a configuration away. The fix is not architectural surgery but honest scoping plus two additions that work today — codeunit 1924 self-profiled scheduled workloads as the interim inside track, and Application Insights telemetry as a third ingestion source and trigger layer — with the fusion claims (event-chain blame percentages) tempered to what 100 ms sampling can statistically support. Add version-aware baselines and a global Microsoft-app cache tier for monthly churn, and the design survives contact with real BC; ship it as written and the first SaaS customer conversation exposes the gap.

## Review 4 — Devil's Advocate (rev-devil)

**1. The plan assumes five existing repos stay frozen while the sixth thing is built. And §1's explicit decision to run both tracks "in parallel" doubles WIP for one person — the most reliable way for a solo project to ship nothing.

**3. (SERIOUS) Zero demand evidence for the platform leap.** The spec has architecture, contracts, phasing, decisions — and not one sentence about a user. The setup cost is enterprise-grade: Docker server, BC extension, per-tenant bearer tokens, a capture daemon next to the BC server, CI pushing source zips + .alpackages on every release (§2), issue-tracker PATs. BC partners with perf pain already have Application Insights telemetry with published KQL schemas, the in-client profiler, and BCPT. The honest differentiator — AL call-tree attribution and pattern detection, which KQL genuinely cannot do — justifies the *analyzer*, which already exists; it does not automatically justify continuous capture + lifecycle + sinks. One design-partner commitment would be worth more than the whole document.

**4. (SERIOUS) Sub-project 3 is a wait state dressed as a workstream.** §Phasing lists "al-perf-bc SaaS enablement" as a parallel track whose first step (scope verification) is already done and confirmed blocked. Everything after is contingent on a third party with no obligation or timeline. Listing it in the phasing table creates the illusion of three parallel progress streams when one of them is "check BC30 artifacts when they drop." It should be labeled a monitored external dependency, not a sub-project.

**5. (SERIOUS) The source registry makes a solo dev the custodian of ISV source code.** §2 has CI pushing full source zips plus symbols, retained per version, with "at-rest encryption option" (§5). For SaaS, that is customer IP — the most guarded asset a BC ISV owns — held by a one-person operation with no legal entity, compliance posture, DPA, or breach story mentioned. No ISV's lawyer approves pushing source to alperf.sshadows.dk. Fusion-in-SaaS is therefore dead regardless of Microsoft's scope decision; the registry's real audience is self-host, where a versioned server-side registry with async build queues is overkill for what could be a directory convention plus content-hash cache (SourceIndexCache already is one).

**6. (SERIOUS) The lifecycle engine will flap on sampling noise, and flapping issues burn trust permanently.** §4 defines resolved (absent N consecutive runs) with reopen-on-reappearance, fed primarily by *sampling* profiles where hotspots drift with load and scheduler timing. Findings oscillating across the resolve threshold generate reopen-comment storms on real GitHub issues watched by real people. Quiet hours and minimum-interval are patches on the symptom; hysteresis tuning on noisy signals is the hard part of every commercial APM alerting product and took them years — the spec gives it one bullet. Auto-filed bot issues get one chance with a repo owner: the first false positive or duplicate ends adoption. Highest trust-risk-to-spec-attention ratio in the document.

**7. (SERIOUS) The agentic ops layer is a liability specced as a feature — three of its four duties don't need an agent.** §4: weekly fleet review is a deterministic report (cron rendering the finding store to markdown); post-fix verification is compare_profiles, which exists, plus a comment template; escalation-into-epics is a grouping query. Only ambiguous-finding triage benefits from a model, and that is a prompt inside the existing --deep pipeline, not a scheduled autonomous agent with write access to customer issue trackers. As specced it adds per-cycle token cost, non-deterministic external writes, an unbudgeted guardrail/eval workload, and MCP tools (findings_list/get/update, report_file, baseline_query) designed today for a system a year+ away. The spec itself rejected "agentic-as-backbone" for non-determinism and token cost (§Architecture) — then re-admits the same risks one layer up.

**8. (SERIOUS) Of six fusion outputs (§3), one is a product, two are payload enrichment, three are demo-ware until proven otherwise.** Runtime-weighted static findings (#3) is the real one — novel, extends existing prioritized_findings, actionable day one. Implicit-trigger attribution (#2) and the AI payload upgrade are legitimate context enrichment, not standalone sections. Event-chain blame's headline metric ("yours is 80% of that fan-out cost") is computed from the profile alone; the static graph only adds never-observed subscribers — a footnote. Fix-leverage ranking (#4) via static fan-in will rank ubiquitous utility helpers at the top on any real BC codebase and needs damping heuristics the spec doesn't mention — high README appeal, low decision value. Edge validation (#5) requires instrumentation captures (OnPrem-only track); fleet heat map (#6) requires a fleet, which requires the blocked capture story. Scoping "the largest chunk" around all six inflates the riskiest sub-project with speculative sections.

**9. (SERIOUS) The phasing back-loads the differentiated value while that value is already sitting on this branch.** Phases 1–2 deliver capture plumbing whose analysis payoff arrives in phase 4. Meanwhile feat/alsem-fusion already contains ~4,100 lines of working fusion machinery with tests — a large fraction of the highest-value sub-project exists at CLI level, unshipped. A value-ordered roadmap ships runtime-weighted findings now and builds capture plumbing when a deployment exists that needs it.

**10. (SERIOUS) The hub is an operational service estate, and the cross-repo version matrix has no owner.** §Architecture's hub = multi-tenant HTTP server + async graph-build queue + SQLite finding store + outbox with sink-ack-gated commits + LRU disk caches + at-rest encryption + PAT secret management + spawned Rust binaries from two other repos. The current /api/ingest is a 191-line POC; the distance to the specced hub is most of a platform team's year. Docker self-host makes every customer environment (their BC version × alsem build × converter build) a support surface. engine-runner.ts checking schemaVersion majors is good discipline, but §5 tests components in isolation — nowhere is a compatibility matrix or release train defined across al-perf × al-call-hierarchy × bc-mdc-converter, which is exactly what breaks when three repos version independently.

**11. (SERIOUS) Instrumentation capture overhead on scheduled production runs is unaddressed.** The outside track's pitch is instrumentation depth ("exact self-times, temporal call tree," §1) on a cron schedule. Instrumentation profiling is not free on the profiled server; a daemon attaching to production BC on schedule is something the customer's sysadmin gets to veto, and the spec offers no overhead numbers, sampling-vs-instrumentation guidance, or blast-radius controls to overcome that veto.

**12. (ANNOYING) Loose ends.** Finding store "in SQLite alongside the existing history store" (§4) = two persistence systems, no migration story, multi-tenant keying grafted onto a local-CLI store. The open-source/SaaS split is undefined — al-perf is published unscoped on npm; if the lifecycle engine ships in the package, SaaS is self-undercut; if not, the split needs design that isn't here. "Status: Approved (brainstorm complete)" on a document locking six sub-projects and a product posture is approval theater.

### The ruthless cut (70% dies)

Keep three things, all in or next to al-perf, no new services:

1. **ir-json ingestion (sub-project 1, unchanged).** Smallest, spec'd, converter exists, upgrades detectors to exact counts, benefits the plain CLI immediately. Highest value per week of anything in the document.
2. **Fusion, CLI-first, cut to two outputs.** Land what's on feat/alsem-fusion: runtime-weighted static findings and static context in AI payloads (publisher/implicit-trigger enrichment for 7.1/7.4). Source acquisition stays --source + local graph paths — the §2 ad-hoc fallback *is* the feature; the registry service is deleted. Event-chain blame only if it falls out of existing EventFlow data for free. Leverage ranking, edge validation, fleet heat map → backlog.
3. **A degenerate lifecycle: fingerprint dedup in the existing history store + a report.** States: new / known / gone. Output: markdown/JSON digest from existing formatters. Sink adapters are replaced by a 20-line documented gh issue create example in the customer's own CI — which also deletes the PAT-custody problem, the outbox, and the flapping risk (their CI, their thresholds, their noise tolerance).

Explicitly parked, not phased: the capture orchestrator becomes a documented cron + bc-dev-mcp + converter one-liner recipe until a named user asks for a daemon; SaaS enablement becomes "blocked on Microsoft, re-check at BC30" (optionally spike SamplingPerformanceProfiler self-capture as a cloud-viable path); the agentic ops layer becomes a deterministic al-perf report digest; ADO sink deleted; hosted SaaS posture reduced to the existing single-profile web flow until a cloud capture path exists.

This preserves every piece of genuinely differentiated analysis and deletes every piece that requires babysitting a service, holding someone's secrets or source, or waiting on Microsoft.

### Steelman — worth keeping regardless

1. **The ir-json interchange contract is the best decision in the document.** A lossless per-invocation IR (spec verified real in bc-mdc-converter) decoupling capture from analysis means every capture mechanism — snapshot debugger, scheduled profiler, whatever Microsoft opens next — reduces to "produce ir-json." It makes the entire capture layer replaceable, which is exactly the right hedge given attack #1.
2. **Runtime-weighted static findings is a real moat.** Nothing in the BC ecosystem joins a whole-program resolved call graph (interface dispatch, event flow, ~0% unknown edges) with measured profile cost; Application Insights structurally cannot. The integration discipline already exists in src/semantic/engine-runner.ts — schema-major validation, content-hash caching, timeout-to-degrade — unusually mature plumbing for an unshipped feature.
3. **The graceful-degradation invariant, especially "store the profile regardless; reanalyzable after a fix" (§5).** Fleet reprocessing after detector upgrades makes the analysis engine retroactively improvable and should be a stated invariant of whatever survives, at any scale from CLI to platform.

**Bottom line:** this is one excellent product (profile analysis with semantic fusion, already ~80% built) wearing the costume of a six-service observability platform whose two capture tracks are both OnPrem-only and whose SaaS premise rests on an unanswered email to Microsoft. Ship the product; archive the costume until someone asks to wear it.

## Review 5 — Future Goals / Strategy (rev-future)

I verified load-bearing contract claims against the repos before reviewing: the ir-json spec (U:\Git\bc-mdc-converter\docs\superpowers\specs\2026-07-06-ir-json-format-design.md) carries an integer schemaVersion with a written versioning policy (its §3.7); al-perf's history store (src/history/store.ts) is JSON-files-on-disk, not SQLite; and src/semantic/engine-runner.ts already major-matches alsem schema versions when spawning the Rust binaries from al-call-hierarchy.

### Findings

1. CRITICAL — §1 Capture / §Phasing (sub-project 3): no hedge for the blocked SaaS capture track. The inside track waits on Microsoft opening OnPrem-scoped objects; the outside track (snapshot debugger, port 7083, Basic auth) is OnPrem-only. Most BC customers are SaaS-hosted, so until Microsoft acts, the "product" half of "product both ways" has no capture story for its main addressable market — and the spec's only plan is waiting. Application Insights telemetry (long-running AL, performance signals, job queue events) is SaaS-viable today, needs no Microsoft permission, and the author already works with it. Recommendation: define ingest as a capture-source adapter seam (payload-type discriminator plus source-kind metadata, not an if/else over two formats) and name an App Insights adapter as the explicit hedge for sub-project 3. Coarser data than profiles, but it keeps the SaaS product path in the author's control.

2. MAJOR — §Phasing: the product-defining layer is sequenced behind the largest technical chunk. The lifecycle engine + sinks (sub-project 5) is what turns al-perf from analyzer into observability product, and it needs nothing beyond today's .alcpuprofile ingestion — yet it sits behind fusion (sub-project 4), explicitly the biggest piece. Fusion enriches findings; the lifecycle doesn't depend on it. Recommendation: pull 5 ahead of (or parallel with) 4. A working capture-to-GitHub-Issue loop on sampling profiles de-risks the product thesis years earlier than fusion does; fusion sections then land as enrichments to an already-running pipeline.

3. MAJOR — §3 vs §4: finding identity is defined after the components that must emit it. Fingerprints (patternId + stableRoutineId + appId + salient location) are specified in sub-project 5, but fusion (sub-project 4) introduces six new finding types that will all need fingerprints. Retrofitting identity onto fusion output types is rework and risks incompatible ad-hoc keys. Recommendation: extract the fingerprint contract (algorithm, fields, explicit fingerprintAlgoVersion) into a standalone contract that lands before or with sub-project 4's output types.

4. MAJOR — §4 Finding identity: fingerprint stability across upgrades is unaddressed. The lifecycle's value rests on fingerprints surviving time, yet a detector rename or a change to how "salient location" is computed would silently resolve and re-file every issue fleet-wide — through the sinks, visibly to users. Recommendation: store the algorithm version with each finding; on algorithm upgrade run a re-fingerprint migration pass linking old to new identities; and guard sink adapters against mass state transitions caused by an algorithm change (the quiet-hours storm guard covers alerts but not this).

5. MAJOR — §2 Source registry / §5 Security: hosted custody of customer source is the design's highest-liability asset and an ISV adoption blocker. The registry stores full source zips plus symbols; for SaaS that is a sales blocker for exactly the ISVs the product targets, and the security section covers encryption at rest but not retention, deletion, or right-to-erasure. Recommendation: leave the door open now for CI-side graph builds — let POST /api/sources accept prebuilt artifacts (tree-sitter SourceIndex + aldump/alsem exports keyed by content hash) as an alternative to raw source, so the hosted service never custodies IP. Add a per-tenant retention/deletion policy and a per-tenant opt-out for sending source to the Anthropic API (capabilities 7.1/7.4).

6. MAJOR — §Component Inventory / missing section: no cross-repo compatibility policy. Six repos, four live contracts (ir-json schemaVersion, alsem inventory/analyze schemas, the /api/ingest API, aldump exports), and no statement of who may break whom. engine-runner.ts's major-match check is good prior art but a local convention, not a platform rule. The ingest API especially needs versioning: orchestrators and al-perf-bc installs in the field will lag the server. Recommendation: add a contracts table to the umbrella doc with the rule "consumer accepts current plus previous major," and version /api/ingest from day one. Also: ir-json — the platform's interchange format — is currently specced inside bc-mdc-converter's repo; once other producers exist (finding 1), it should be treated as platform-owned.

7. MAJOR — missing risk register: both capture tracks depend on Microsoft, and Microsoft is also the Sherlocking threat. The outside track rides undocumented internals (.mdc format, snapshot debugger endpoint) that any BC release can break or close; the inside track rides a pending scope decision; and Microsoft keeps improving native telemetry and the in-client profiler. None of this is named as risk. Recommendation: add a risk register with mitigations — .mdc breakage per BC release (converter version gate exists; raw .mdc is already retained per §5), scope request denied (finding 1's App Insights hedge), native Sherlocking. Be explicit about what's defensible: cross-app semantic fusion (Microsoft has little incentive to blame ISV event subscribers with whole-program graphs) and the lifecycle-to-GitHub/ADO loop. Plain hotspot views are commodity and will be Sherlocked — don't invest differentiation there.

8. MAJOR — §Product posture: SaaS corners in the single-process design, and no metering. SQLite plus one Bun process is right for self-host and a solo dev, but registration-triggered graph builds spawn memory-hungry Rust processes over customer source in the same process space as ingest — the noisy-neighbor problem arrives with tenant number two. There are no per-tenant quotas and no usage counters, and billing cannot be retrofitted without metering data. Recommendation: do not adopt Postgres now; instead keep storage behind narrow repository interfaces, route analysis and graph builds through a queue abstraction (in-process implementation is fine), bound concurrent alsem/aldump spawns, and record per-tenant ingest/analysis counters from the first release — cheap now, enables billing later.

9. MINOR — deployment: binary distribution across three toolchains is unstated. aldump/alsem (Rust) must ship inside al-perf's Docker image; bc-mdc-converter (Rust) ships with the orchestrator; multi-arch builds and version pinning are nobody's spec. Recommendation: publish the Rust binaries as checksummed release artifacts, pin versions in the Dockerfile, and extend the existing schemaVersion check into a startup health handshake.

10. MINOR — §1 data contracts: no pprof/OTLP consideration. ir-json is well-versioned, but the OpenTelemetry profiling signal is maturing and pprof unlocks existing viewers (speedscope, Pyroscope) for free. Recommendation: add a short appendix to the ir-json spec mapping its fields to pprof semantics — the Firefox-profile export in bc-mdc-converter shows the pattern. Keeps interop doors open at near-zero cost.

11. MINOR — §4 sinks: define the adapter interface, not just the first two adapters. Deferring Slack/Teams/email is right, but v1 should specify SinkAdapter as an interface owning the outbox/retry/idempotency semantics, so later sinks are additive files rather than a refactor. The spec reads as if this is intended — say it explicitly.

12. MINOR — §4 storage: two persistence mechanisms. The new SQLite finding store lands "alongside" the existing JSON-file history store (src/history/store.ts), leaving two stores to tenant-key, encrypt, and back up. Recommendation: migrate history into the same tenant-keyed SQLite when the lifecycle engine lands.

13. MINOR — §4 agentic ops: the resilience posture is right; make its invariants explicit. Deterministic core with an agentic shell writing through the same sink adapters is exactly the model-churn-resistant shape — findings never depend on the agent. Strengthen it: state that invariant in the spec, version the agent's prompts in-repo, add a per-cycle token budget to config, and require bring-your-own-key for self-host so the layer survives model and pricing churn. MCP churn risk is already contained by the "MCP tools are thin wrappers" principle.

### Verdict

The architecture is strategically sound where it matters most: the hub-with-deterministic-core shape, the lossless versioned ir-json contract, graceful degradation at every optional seam, and the agentic layer as shell rather than backbone are all decisions that will age well over 2-3 years, and the existing code shows the contract discipline (schema major-matching, content-hash caching) is real rather than aspirational. The gaps concentrate on the product half of "product both ways": the spec has no capture hedge under the author's own control while both tracks wait on Microsoft, it sequences the product-defining lifecycle engine behind the largest research-flavored chunk, and it leaves the three genuine product blockers — customer source custody, cross-repo compatibility policy, and tenant metering — unstated rather than deferred deliberately. None of this requires redesign; it requires reordering the phases (lifecycle before fusion), one new adapter seam (App Insights), and about a page of added contracts (fingerprint versioning, compat matrix, risk register). Done now, all are cheap; done in year two, the fingerprint and registry decisions in particular become migrations against live customer data.

## Review 6 — Design Coherence (rev-design)

I verified the spec's claims against the actual code in al-perf (feat/alsem-fusion), al-call-hierarchy, bc-mdc-converter, bc-dev-mcp, and al-perf-bc.

### Critical

**1. Encryption-at-rest contradicts reanalyzability and fleet reprocessing.** (§2 Ingest, §5 Error handling) — The POC path the spec says is "kept for SaaS mode" encrypts the profile with the *tenant's* public RSA key and deletes the plaintext (U:\Git\al-perf\web\handlers\ingest.ts:118-150, "v1 invariant: only ciphertext at rest"); the server holds no private key. §5 simultaneously promises "the profile is stored regardless and is reanalyzable after a fix — this also enables fleet reprocessing after detector upgrades," and §2's "queue for analysis" implies plaintext must survive until an async worker runs. All three cannot hold in SaaS mode. **Recommendation:** decide explicitly — either a server-held envelope key (KMS-style) for SaaS so the server can decrypt for reprocessing, or scope reanalysis/reprocessing/late-fusion to self-host mode only, and specify plaintext lifetime for the async analysis queue either way.

**2. Multi-app merged-workspace graphs contradict "build on registration, not on first profile."** (§2 Source registry, Graph cache invariants) — Registration builds are per {appId, version}, but §2 also says "the graph is built over the merged workspace" for profiles spanning base app + ISV apps. The app *combination* is only known when a profile arrives, so a merged graph cannot be built at registration; the two invariants conflict, and combination-graph identity, build trigger, and eviction are all undefined. **Recommendation:** define a combination-graph key (hash of the sorted {appId, version} set), build it lazily on first profile with that combination (profile-only analysis until ready), and state how per-app registration artifacts (symbols, SourceIndex) feed the merged build.

### Major

**3. The fusion-join description doesn't match the implemented contract.** (§3) — The spec says fusion "joins them on routine identity: al-perf's stableRoutineId on one side, alsem fingerprints on the other." The actual join (U:\Git\al-perf\src\semantic\correlate.ts:6-8) is name-based — (canonicalObjectType, objectNumber, normalizeTriggerName(routineName)) — with explicit confidence buckets (matched / ambiguous / blind-spot / cold / unkeyable). stableRoutineId is *emitted by alsem's inventory* and adopted after correlation, and "alsem fingerprints" are finding-level identities, a different space. As written, the spec conflates two identity spaces and hides the ambiguity/blind-spot machinery that everything downstream (finding fingerprints, lifecycle) must handle. **Recommendation:** restate the join accurately and require the lifecycle engine to define behavior for ambiguous and blind-spot correlations (e.g., no stableRoutineId-based fingerprint for ambiguous matches).

**4. The finding fingerprint formula is undefined for the profile-only path.** (§4 Finding identity) — Fingerprint = hash(patternId + stableRoutineId + appId + salient location), but stableRoutineId exists only when an alsem inventory matched. The spec's own graceful-degradation principle means many findings (unregistered sources — the common case early on) have no stableRoutineId, so most lifecycle findings are un-fingerprintable by the given formula. "Salient location" is also undefined, and line conventions differ between capture formats (ir-json emits 0-based raw wire lines per the bc-mdc-converter spec §3.5; .alcpuprofile uses display lines) — the same finding could fingerprint differently by capture kind. **Recommendation:** define a fallback identity key (the correlate join key: appId + canonical objectType + objectNumber + normalized routine name), define fingerprint migration when a source registers later, and pin "salient location" to a normalized, capture-kind-independent convention.

**5. Rename/signature instability of stableRoutineId is unacknowledged.** (§4 Finding identity) — stableRoutineId = stableObjectId + normalizedSignatureHash(name, params, returnType) (U:\Git\al-call-hierarchy\src\engine\snapshot.rs:191-195). It survives line shifts and body edits, but renaming a routine or changing its signature severs the fingerprint: the old finding silently auto-resolves after N runs and a duplicate issue is filed with no history. The spec claims only line-shift tolerance, which is true but sidesteps the case that generates user-visible noise under the lifecycle design. **Recommendation:** acknowledge the limitation in §4 and specify a mitigation — alsem's differential machinery as prior art for rename detection, or a manual fingerprint-merge operation exposed via findings_update.

**6. "Run" and absence are undefined — resolution breaks across mixed capture types.** (§4 Lifecycle states) — "resolved (absent N consecutive runs)": absent from *what*? A sampling capture of a different user activity never exercises the routine; instrumentation-only findings (edge validation, exact-count detectors) can never appear in sampling runs. Without scoping, findings resolve spuriously whenever capture mix or user activity shifts. **Recommendation:** define run = a capture from the same (tenant, schedule/job) stream; count absence only in runs of a compatible capture kind that actually exercised the containing app/object (profile contains frames in that app), and write this definition into §4.

**7. §5 contradicts §4 on sink failure.** — §4: "finding state transitions commit only after sink acknowledgement." §5: "Sink failure: outbox queue with retry." An outbox pattern by definition commits state locally and delivers asynchronously; ack-gated transitions mean a GitHub outage freezes the lifecycle engine — and many transitions have no sink action at all (trigger-suppressed findings, quiet hours). **Recommendation:** commit transitions locally, treat sink delivery as separate outbox rows with retry, and let issues converge eventually; drop the ack-gating sentence.

**8. Lifecycle state machine is incomplete.** (§4) — needs-triage (referenced in the agentic ops section) is not a state or flag in the machine. Transitions into/out of improving, from regressed back to open/improving, and from new to open are unspecified; whether re-appearance after closed (vs resolved) reopens or files fresh is unstated; and which transitions the agent may perform via findings_update (can it force resolved, conflicting with absence-based resolution?) is undefined. **Recommendation:** replace the one-line arrow diagram with a transition table (state × event → state, with guards), including needs-triage and an explicit list of agent-permitted transitions.

**9. "Auth: bearer per tenant (exists)" is inaccurate.** (§2 Ingest, §5 Security) — The implementation uses one shared global secret (U:\Git\al-perf\web\poc-secret.ts:18-26, checked at web/handlers/ingest.ts:26) with the tenant chosen by an x-tenant-id header; any bearer holder can write into any tenant's store. §5's "tenant isolation: bearer per tenant (exists)" therefore doesn't hold today. **Recommendation:** mark per-tenant tokens as new work (issue them at /api/tenants/register) and assign it to a sub-project — it's currently in nobody's scope, yet SaaS multi-tenancy depends on it.

**10. Graph-build failure and mid-build profiles are unhandled.** (§2, §5) — §5's error-handling list covers capture, analysis, sink, and no-match — but not the async registration build failing (aldump/alsem crash, bad symbols), partial success (tree-sitter SourceIndex OK, alsem fails), how failure is reported to the CI caller of POST /api/sources, or what happens to profiles arriving while a build is in flight. **Recommendation:** add registry entry states (pending / built / partial / failed) with a pollable status API, profile-only analysis until built, partial degradation (source correlation without fusion) when only alsem fails, and a policy on re-fusing profiles that arrived during the build window (constrained by finding 1 in SaaS mode).

**11. Duplicate-ingestion and replay semantics stop at storage; the lifecycle layer needs event-time ordering.** (§2, §4) — "Idempotency-key dedup (exists)" overstates the code: the handler validates the key and uses it as a directory name, then unconditionally re-analyzes and overwrites (web/handlers/ingest.ts:36-40, 93-133) — a duplicate POST re-triggers the whole pipeline, including lifecycle evaluation once that exists. Separately, fleet reprocessing of *old* profiles after detector upgrades (a stated feature) replays findings out of temporal order, which can resurrect resolved findings or corrupt "absent N runs" counting. **Recommendation:** true dedup (no-op response on a seen key), and lifecycle evaluation keyed to profile capture time (event time) and idempotent per (fingerprint, profileId).

### Minor

**12. Nearest-lower version fallback is unbounded.** (§2 Graph cache invariants) — Falling back from 25.1 to 24.0 silently analyzes against substantially different code; a warning flag alone doesn't bound the error, and per-app independent fallback in multi-app profiles compounds it. Bound fallback (same major, configurable) and record the actually-matched version in the result and in finding fingerprint provenance.

**13. Formatter-parity cost of six new sections is unstated.** (§3) — Fusion adds ~6 section types to AnalysisResult plus a batch section (fleet heat map — which the spec should explicitly place in BatchAnalysisResult); the compile-time SectionRenderers<T> mechanism forces renderers in all four formatters for each. Correct design (parity honored), but sub-project 4's scope should acknowledge the ×4 rendering work or consciously stage it.

**14. Regression baselines must be per capture kind.** (§4 Trigger rules) — Sampling statistical self-time and ir-json exact selfTicks (plus instrumentation overhead) are not comparable; "routine self-time +X% vs a rolling baseline" produces false regressions when a schedule mixes kinds. Key baselines by (tenant, schedule, captureKind).

**15. No packaging/version-pinning story for the three spawned binaries.** — The hub spawns alsem/aldump (schema pinned at 1.1.0 in src/semantic/contracts.ts:17-20) and the orchestrator spawns bc-mdc-converter (ir-json schemaVersion 1). Nothing says how Docker/SaaS images acquire correct binary versions or what happens on a schema mismatch at fusion time. Add a packaging note and an ir-json schemaVersion pin mirroring the alsem EXPECTED_*_SCHEMA_VERSION pattern.

**16. Two fingerprint namespaces, one formula.** (§3 output 3, §4) — Runtime-weighted alsem findings already carry their own carefully designed fingerprints (contracts.ts FindingSummary); the spec's single hash formula implies re-fingerprinting them. State the mapping — e.g., alsem-originated findings keep their native fingerprint under an alsem: namespace, pattern findings use pattern:<hash> — so lifecycle identities never collide across origins.

**17. Testing section misses the highest-risk integration seams.** (§5) — No tests listed for registry version matching / nearest-lower / multi-app merge, no contract-pin test for ir-json schemaVersion, no inside-track E2E (the POC roundtrip script scripts/poc-roundtrip.ts exists and should be referenced), and nothing for the agentic layer (tool-contract tests at minimum). The docker-compose E2E covers only the outside track.

**18. Partial/incomplete captures are unaddressed at platform level.** (§1) — ir-json explicitly models isIncomplete/incompleteCount and the orchestrator has duration/stopCondition, but the umbrella never says whether an incomplete capture is analyzed normally, flagged, or excluded from lifecycle run-counting (interacts with finding 6). Also §1 never states the outside track's *sampling* output format (implied .alcpuprofile).

**19. Relationship of the new SQLite finding store to the existing JSON history store is unspecified.** (§4) — The existing history store is per-file JSON (src/history/store.ts), and §4's regression baselines need history data. Two parallel stores with overlapping content invite drift; state whether history/baselines migrate into SQLite or stay separate with a defined owner for baseline computation.

### Verdict

This is a strong umbrella spec where it describes what exists: the component inventory is accurate against all five repos (I verified /api/ingest, the engine-runner/alsem integration, ir-json's drafted format spec at U:\Git\bc-mdc-converter\docs\superpowers\specs\2026-07-06-ir-json-format-design.md, aldump/EventFlow/ImplicitTrigger/integration-points in al-call-hierarchy, sampling profiles carrying appId/appVersion in frames for registry auto-match, and the Microsoft scope request in al-perf-bc), the hub-plus-agentic-ops architecture decision is well-reasoned, degradation principles are consistently applied, and the phasing is genuinely parallelizable. It is weakest exactly where it is newest: the two criticals are real internal contradictions (SaaS encryption vs reprocessing; per-app registration builds vs merged-workspace graphs) that will silently shape sub-projects 4 and 5 if not resolved at the umbrella level, and §4's lifecycle engine — finding identity for the degraded path, rename tolerance, the meaning of a "run", and the sink-ack/outbox conflict — is under-specified relative to how load-bearing it is for everything above it. I recommend one revision pass focused on §2's cache invariants, §4's identity and state machine, and reconciling §5's encryption and sink-failure text before the sub-project 4 and 5 specs are drafted; sub-projects 1–3 can proceed on the current text.
