# Continuous Monitoring — POC Scope

**Date:** 2026-04-13
**Companion to:** [2026-04-13-continuous-monitoring-design.md](./2026-04-13-continuous-monitoring-design.md) (rev 2)
**Purpose:** Explicit "build this, skip these" cut list for proof-of-concept builds. Reduces full design scope to fastest demonstrable subset.

## Why this doc exists

Rev 2 design covers the full Phase 1 production target. POC needs a smaller cut — enough to demonstrate value, not enough to ship. This doc:
- Lists the minimum viable scope at three staging levels (v0, v1, v2).
- Cross-references each item to the section in the rev 2 spec it implements (or skips).
- Calls out POC limitations explicitly so they don't accidentally ship to customers.

## Staging levels

| Level | Outcome demonstrated | Effort estimate | Audience |
|---|---|---|---|
| **v0 — "it flows"** | Profile auto-ships from BC to al-perf, gets analyzed, viewable. No encryption, no real auth. | 5–7 days | Internal validation; de-risk BC Performance Profiles read path |
| **v1 — "encrypted flow"** | v0 + tenant-controlled keypair, sensitive payload encrypted at rest at al-perf. Manifest hash binding. BC decrypts on Open Profile. | +5–7 days | Customer / stakeholder demo. The headline pitch. |
| **v2 — "production-shaped"** | v1 + HMAC request signing, bootstrap tokens, atomic ingest publication, idempotency, compression, BYO Anthropic key. | +7–10 days | Pilot customer onboarding |

**Recommended path:** v0 → v1 → v2, in order. Each builds on the prior. v1 is the smallest that's demo-worthy externally.

## POC v0 — "it flows" (5–7 days)

**Goal:** Schedule profiler in BC, wait 5 min, see analysis appear in al-perf web. Single tenant. No security. Internal-only.

### BC side build (al-perf-bc)

| Item | Spec section | POC simplification |
|---|---|---|
| Setup table `AL Perf Ship Setup` | "Components → BC side → Table 70503" | 4 fields only: `Enabled`, `Tenant Code`, `Server URL Base`, `Last Run DateTime`. No filters, no skip counters, no key version. |
| Shared secret | "Auth & identity lifecycle" | One hardcoded constant in IsolatedStorage. No HMAC. Sent as `Authorization: Bearer <secret>` header. Document as POC-only. |
| Ship log table `AL Perf Ship Log` | "Components → BC side → Table 70504" | 3 fields: `Activity ID` (PK), `Status` (Pending/Shipped/Failed), `Error Message`. No retry counts, no key version, no skip reason. |
| Job Queue codeunit `AL Perf Auto Ship` | "Per-profile ship" | Window = `Starting Date-Time >= LastRun`. No look-back, no overlap. For each unsent profile: `CalcFields(Profile)`, POST plaintext to `/api/ingest`, mark Shipped on 202. No retry on failure. |
| Manual setup of Job Queue Entry | (deployment) | Admin creates Job Queue Entry pointing at codeunit, frequency 5 min. |
| Open Profile action on Ship Log | "Decryption (BC side)" | GET `/api/profiles/{activityId}` returns plaintext blob. Pipe to existing `Sampling Performance Profiler.SetData()`. No decryption. |

**SKIP:** keypair generation, encryption codeunit, key ring, backup, compression, page extensions on 1931/1933, schedule filter, activity type filter, min duration, daily limit, status refresh.

### al-perf web side build

| Item | Spec section | POC simplification |
|---|---|---|
| `POST /api/ingest` | "Server-side processing" | Validate constant bearer secret. Multipart parse. Save raw `profile.bin` + `manifest.json` to `web/storage/{tenantCode}/profiles/{activityId}/`. Run `analyzeProfile` synchronously inline (no queue, no worker). Save `metrics.json` and `result.json` (plaintext). Return 202 + `{id, status:"stored"}`. |
| `GET /api/profiles/{activityId}` | "Decryption (BC side)" step 1 | Returns `profile.bin` bytes (plaintext). |
| Tenant store | "Tenant store" | Single `web/tenants/poc.json` with hardcoded `tenantCode` and shared secret. No KEK. |
| Path validation | "Identifiers and validation" | Hardcode tenantCode = constant. No regex needed since not user-controlled. Still validate `activityId` regex (free, prevents path traversal). |

**SKIP:** registration endpoint, secret/key rotation, public key store, idempotency record, worker, DLQ, status endpoint, history/trend endpoints, KEK envelope, log scrubbing infrastructure (just don't log Authorization header).

### What v0 proves

- BC `Performance Profiles` table read works as expected (the platform-table feasibility risk).
- Job Queue Entry triggers codeunit on schedule.
- `Profile` BLOB CalcFields + InStream + multipart upload works.
- Existing al-perf `analyzeProfile` consumes the bytes correctly.
- Existing `Sampling Performance Profiler.SetData()` round-trips a downloaded profile.

### POC v0 limitations (label loudly)

- No encryption — anyone with disk access reads profiles.
- Single tenant — secret hardcoded.
- No retry, no idempotency — duplicate ships re-process.
- Synchronous analyze — large profile blocks ingest endpoint.
- DO NOT EXPOSE TO INTERNET. Localhost / private network only.

## POC v1 — "encrypted flow" (+5–7 days on top of v0)

**Goal:** Server admin doing `cat blob.enc` sees only ciphertext. BC decrypts on Open Profile using private key from IsolatedStorage. Demonstrates the headline differentiator.

### Add to BC side

| Item | Spec section | POC scope |
|---|---|---|
| Codeunit 70504 `AL Perf Crypto` (subset) | "Codeunit 70504" | Implement `GenerateKeypair`, `GetCurrentPublicKeyXml`, `GetPrivateKeyForVersion(var SecretText): Boolean`, `DecryptBundle`. Skip `ExportEncryptedBackup` / `ImportEncryptedBackup`, skip `SignRequest` (still using bearer). |
| Single key version | "Public key rotation" | Hardcoded `keyVersion = 1`. IsolatedStorage key `'al-perf-private-key'` (no version suffix). |
| Setup card actions | "Pages" | "Generate Keypair" + "Register" actions. Register POST sends `{publicKeyXml, tenantCode, sharedSecret}` (POC), not bootstrap-token model. |
| Open Profile decrypt path | "Decryption (BC side)" steps 2–7 | Full implementation. RSA-OAEP unwrap → split K_enc/K_mac → manifest hash → HMAC verify → AES decrypt → SetData. |

### Add to al-perf web side

| Item | Spec section | POC scope |
|---|---|---|
| `POST /api/tenants/register` | "Bootstrap token" | POC simplification: accept `{tenantCode, publicKeyXml, sharedSecret}`. Store pubkey JWK. No bootstrap token. Refuse if `tenantCode` already registered. |
| `web/crypto.ts` | "Crypto module" | Implement `xmlRsaToJwk`, `encryptBundle`. Skip `verifyAuthHeader` (still bearer), skip `wrapTenantSecret` (plaintext in tenant.json — POC limit). |
| Worker | "Worker" | Pull from in-memory queue (still inline-after-ingest acceptable for POC if request ≤ a few MB). `analyzeProfile`, `encryptBundle`, write `wrapped.bin`, `blob.enc`, `result.enc`, `keyversion.txt`, `manifest.json` (plaintext sidecar — required for hash binding), `metrics.json`. |
| `GET /api/tenants/{id}/profiles/{activityId}` | "Decryption (BC side)" step 1 | Returns full bundle as documented. |

### Critical correctness — must not skip

- **Manifest plaintext sidecar.** Server stores byte-exact `manifest.json` separate from `metrics.json`. BC reads, hashes for HMAC verify. Skipping this = HMAC always fails. Non-negotiable.
- **`keyversion.txt`.** Even with single version, write it. Forward-compatible with v2's rotation.
- **Encrypt-then-MAC over IV || manifestHash || ciphertext.** Documented byte layout, no shortcuts.
- **`var SecretText` out parameter.** Don't try `[TryFunction]` returning SecretText.

### What v1 proves on top of v0

- AL `RSA` + `Rijndael` + `Cryptography Management` codeunits work in the BC sandbox (the AL Cloud-feasibility risk for the encryption stack).
- Cross-language crypto contract: Node-side encrypt + AL-side decrypt round-trips correctly.
- Manifest binding works as spec'd.
- BC decrypted profile renders in existing Performance Profiler page.

### POC v1 limitations

- Bearer secret in plaintext header = anyone capturing one request can replay forever. Demo on TLS only, mention as roadmap item.
- No bootstrap token = anyone reaching `/api/tenants/register` first claims a tenantCode.
- KEK absent = `cat web/tenants/{id}.json` shows shared secret. Document.
- Single key forever — if private key lost, all profiles unreadable. No backup format.
- No size limits — large profile may blow Rijndael Base64 ceiling. Test with realistic profile sizes; if hit, add v2 compression earlier.
- No idempotency — duplicate ship re-encrypts and overwrites.
- No status polling — BC marks Shipped on 202; if worker fails inline, admin sees nothing. Inspect server logs.

## POC v2 — "production-shaped" (+7–10 days on top of v1)

**Goal:** Demo to pilot customer. Auth and reliability story holds up to the obvious questions.

### Add to BC side

| Item | Spec section | Scope |
|---|---|---|
| HMAC request signing | "Request signing" | Implement `AL Perf Crypto.SignRequest`. Replace bearer header with `Authorization: HMAC-SHA256 keyId=...,ts=...,nonce=...,sig=...`. |
| Bootstrap token registration flow | "Bootstrap token" | Operator pastes token from CLI. POST register without secret in body. Receive secret in response, store IsolatedStorage. |
| Compression | "Compression and chunking" | gzip via `Codeunit "Data Compression"` before encrypt. Set `X-Profile-Encoding: gzip`. |
| BYO Anthropic key | "Auth & identity lifecycle" | Setup card "Set Anthropic Key" action → IsolatedStorage. Send as `X-Anthropic-Api-Key` header on ship requests when set. |
| Refined retry | "Per-profile ship" step 5 | 4xx terminal except 408/429; 5xx + connection retried. Stop when age >= 6 days OR attempts >= 5. |
| Status field split | "Table 70504" | Status = `Accepted`/`Stored` distinction; refresh action polls server status endpoint. |

### Add to al-perf web side

| Item | Spec section | Scope |
|---|---|---|
| Bootstrap token CLI + store | "Bootstrap token" | `al-perf admin issue-token` writes `web/bootstrap-tokens/{SHA-256(token)}.json`. **Atomic claim via `O_EXCL`** — required, no shortcut. |
| HMAC verification + replay cache | "Request signing" | In-memory replay cache keyed `(tenantCode, nonce)` with 10-min TTL. Reject ts skew > 5 min. Sign over METHOD || PATH+QUERY || ts || nonce || SHA-256(body). |
| Atomic `_inflight/` write | "Server-side processing" step 6 | Write to `_staging/{activityId}.partial/`, fsync, write idempotency record `state:"queued"`, then atomic `rename` to `_inflight/{activityId}/`. Worker watcher ignores `*.partial`. |
| Idempotency record (minimal) | "Server-side processing" step 4 | Two states: `queued`/`done`. Cached response replayed on duplicate `X-Idempotency-Key`. Skip `processing`/`failed` for v2 (no DLQ yet). |
| Atomic idempotency reservation | (rev 2 review HIGH #4) | Exclusive-create idempotency file with `state:"receiving"`. EEXIST → load + return cached. Closes check-then-act race. |
| Decompression | "Compression and chunking" | If header `gzip`, decompress before analyze. |
| Size limits | "Size limits" | Reject 413 if Content-Length > 50 MB or per-part exceeded. |
| Worker (out-of-line) | "Worker" | Filesystem-watch on `_inflight/`, single-instance daemon. Encrypt, persist, advance idempotency to `done`. On exception: log + leave record stuck (DLQ deferred). |
| `GET /api/tenants/{id}/profiles/{activityId}/status` | "Endpoints" | Returns idempotency record state. BC polls for Accepted → Stored advance. |
| Anthropic key handling | "Log scrubbing requirements" | Header → `_inflight/anthropic.tmp` (chmod 0600 on Linux) → worker reads as `Buffer`, `unlink` immediately, `.fill(0)` after analyze. NEVER stored in idempotency record. Reverse proxy + app log scrubbing for `Authorization` and `X-Anthropic-Api-Key`. |
| Pin Node `oaepHash:'sha1'` | (rev 2 review MEDIUM #6) | Explicit in `crypto.publicEncrypt({padding: RSA_PKCS1_OAEP_PADDING, oaepHash: 'sha1'}, ...)`. Prevents Node-version drift. |

### Still SKIP for v2

- KEK envelope (Phase 2 production)
- Public key rotation (single key version, document upgrade path)
- Tenant secret rotation endpoint (Phase 2)
- Encrypted backup blob (Phase 2 — operationally documented "lose private key = lose history" warning suffices for v2)
- Worker stuck-processing recovery scan (Phase 2 — single-instance trust + manual cleanup)
- DLQ (Phase 2 — exception log + manual)
- `/api/tenants/{id}/history` + `/api/tenants/{id}/trend` (separate dashboard spec consumes these)
- Per-tenant rate limiting (Phase 2)
- AL constant-time HMAC compare (Phase 2 hardening)
- Streaming chunked encryption for >8 MB compressed (Phase 2)

### What v2 proves on top of v1

- Replay-resistant auth: capturing one request doesn't enable replay.
- Atomic durable acceptance survives kill-server-mid-ingest test.
- Idempotency: replay same `X-Idempotency-Key` 5× → 1 worker invocation.
- Compression: realistic 5–10 MB profile fits within size budget.
- BYO Anthropic key flows end-to-end without leaking to logs.

### POC v2 limitations

- Single-instance worker — kill it mid-processing, idempotency record stuck `processing` (since deferred). Restart = manual `_inflight/` cleanup. Document.
- No KEK = tenant secrets in plaintext on disk. Document.
- No public key rotation = if private key compromised, full re-registration with new tenantCode. Document.
- No encrypted backup = lose private key = lose history. Loud warning at keypair generation.

## Cut list summary (what's deferred to Phase 1 production after POC v2)

These are spec items deliberately NOT in any POC level. Build them when graduating from POC to Phase 1 production:

1. **KEK envelope at server** — encrypt tenant secrets with master key.
2. **Public key rotation** — versioned ring, monotonic enforcement, BC private key ring.
3. **Tenant secret rotation endpoint** — PATCH with HMAC-signed.
4. **Encrypted backup blob** — `ALPRBK1` format, PBKDF2 + AES + HMAC. Pure AL.
5. **Worker stuck-processing recovery** — startup scan, lease timestamps, requeue stale.
6. **DLQ** — `_dlq/` directory, error logs, BC visibility via status endpoint.
7. **Read APIs** — `/history`, `/trend`. Consumed by separate dashboard spec.
8. **Per-tenant rate limiting** — token bucket.
9. **AL constant-time HMAC compare** — hardening.
10. **Streaming chunked encryption** — >8 MB compressed payloads.
11. **Worker daemon supervision** — systemd / pm2 / Bun supervisor docs.
12. **HSTS + cipher suite policy** — server-side TLS hardening.
13. **Activity Type Filter "All"** — enumextension or wrapper-bool implementation pattern.

## POC build order

Parallel where possible.

### v0 (5–7 days)
1. (BC, 1 day) Setup table + ship log table + setup card.
2. (BC, 1 day) Job Queue codeunit — read Performance Profiles, build multipart, POST.
3. (BC, 0.5 day) Open Profile action — GET + SetData.
4. (web, 0.5 day) `/api/ingest` plaintext, save to disk, inline analyze.
5. (web, 0.5 day) `/api/profiles/{id}` plaintext return.
6. (e2e, 1 day) Wire up, test on BC sandbox.
7. (slack, 1 day) Buffer for unknowns (BC sandbox setup, Job Queue Entry config quirks).

### v1 (+5–7 days)
1. (BC, 1.5 days) `AL Perf Crypto` codeunit subset — keypair gen, encrypt helpers, decrypt helpers. Cross-language unit tests against Node side.
2. (BC, 0.5 day) Setup card "Generate Keypair" + "Register" actions.
3. (BC, 0.5 day) Open Profile decrypt path replaces plaintext path.
4. (web, 1 day) `web/crypto.ts` xmlRsaToJwk + encryptBundle. Worker encrypt step.
5. (web, 0.5 day) `/api/tenants/register` POC version.
6. (web, 0.5 day) Bundle response shape + manifest sidecar persistence.
7. (e2e, 1 day) Cross-language round-trip verification with realistic profile.
8. (slack, 1 day) Buffer for `.NET RSA XML → JWK` parsing edge cases.

### v2 (+7–10 days)
1. (web, 1.5 days) HMAC verifier + replay cache + bootstrap token CLI + atomic claim.
2. (BC, 1 day) `SignRequest` in `AL Perf Crypto` + bootstrap-token registration UI.
3. (web, 1.5 days) Atomic `_inflight/` write + idempotency reservation + out-of-line worker.
4. (BC, 1 day) Compression via Data Compression codeunit; gzip flag.
5. (web, 0.5 day) Decompression in worker; size limits.
6. (BC + web, 1 day) BYO Anthropic key flow + log scrubbing test.
7. (BC, 0.5 day) Status field split + refresh poll action.
8. (e2e, 1 day) Replay attack + kill-mid-ingest tests.
9. (slack, 2 days) Buffer for surprises in HMAC cross-language consistency, atomic-rename quirks.

## Definition of done

- **v0:** Profile shipped automatically within 5 min, `analyzeProfile` result viewable via `/api/profiles/{id}`. No regressions in existing manual `/api/analyze` path.
- **v1:** Server admin `cat web/storage/{tenant}/profiles/{id}/blob.enc` shows ciphertext. BC Open Profile decrypts and renders. Manifest hash binding test passes.
- **v2:** Capture + replay test fails (401 due to nonce dedup or ts skew). Kill-server-mid-ingest test: BC retries successfully on next run, no duplicate worker invocation. Compression: 5 MB profile compresses to <1 MB on wire, full round-trip works.
